## What is a JPA Entity, and why `@Entity`?

A JPA **Entity** is a Java class that represents one row in one database table.
`@Entity` + `@Table(name = "...")` tell Hibernate "map this class to that table."
Each field = one column. `@Id` marks the primary key field.

Two constructors are typical: a `protected` no-arg one (required by Hibernate
internally to reconstruct objects when loading from the DB) and a real one
your own code calls (e.g. `new ReviewTask("...")`).

`@PrePersist` methods run automatically right before a row is first saved —
used here to auto-fill timestamps like `createdAt`.

Relationships (e.g. a `KnowledgePoint` belonging to a `ReviewTask`) are expressed
with `@ManyToOne` + `@JoinColumn`, storing a reference to the actual Java object
instead of a raw foreign-key number — Hibernate converts between the two.

See: `backend/src/main/java/com/examreview/examrevieworganizer/entity/`

## What is Hibernate?

**JPA** is just a specification (a contract/set of annotation rules) — it does
nothing by itself. **Hibernate** is the actual library that implements that
contract: it reads `@Entity` classes, generates real SQL, sends it to
PostgreSQL, and converts result rows back into Java objects.

JPA = job description. Hibernate = the employee actually doing the job.
(EclipseLink is another JPA implementation, but Hibernate is what Spring Boot
uses by default, and what we're using.)

## Backend / Frontend / PostgreSQL — who does what

Three separate running things, only two of which we write code for:

```
React app (browser)  →  Spring Boot app (server)  →  PostgreSQL (database)
   we write this           we write this              already-installed software
   (frontend/)              (backend/)                 (we configure, don't code)
```

PostgreSQL actually stores data permanently on disk and enforces rules (like
foreign key constraints) at the database level, independent of the Java app.

"We don't write SQL" means Hibernate generates SQL *for us* — SQL still
happens, it's just auto-generated instead of hand-typed. Proven by watching
Hibernate print `insert into review_tasks (...) values (...)` in the app logs,
then seeing that exact row show up when querying PostgreSQL directly via
`psql`, completely independent of the Java app.

Restaurant analogy: PostgreSQL = kitchen, Hibernate = waiter who speaks the
kitchen's language (SQL), our Java code = us deciding what to order, React =
the dining room (never talks to the kitchen directly).

## What does `ExamReviewOrganizerApplication.java` do?

Auto-generated by Spring Initializr on Day 1 (not something we wrote by hand).
It's the entry point of the whole backend — the one file with `main()`.

- `@SpringBootApplication` — shorthand for: scan this package + subpackages for
  our classes, auto-configure things like the embedded web server and DB
  connection (based on `application.properties`), and mark this as a
  Spring-managed app.
- `SpringApplication.run(...)` — the line that actually boots everything:
  starts Tomcat, connects to PostgreSQL via Hikari, discovers `@Entity` classes
  and `JpaRepository` interfaces, wires it all together.

Normally stays small and untouched — business logic belongs in
Controller/Service/Repository classes, not here.

## `@GeneratedValue(strategy = GenerationType.IDENTITY)`

Pairs with `@Id` to say "the database generates this primary key, not me."
`IDENTITY` specifically means: rely on Postgres's own auto-increment column
(`GENERATED BY DEFAULT AS IDENTITY`, the modern replacement for `SERIAL`).

Proof: in a demo save, `saved.getId()` was `null` before `.save()`, and became
`1` right after — Hibernate ran the `INSERT`, then Postgres handed back the id
it generated.

Other strategies: `SEQUENCE` (separate DB sequence object, lets Hibernate
batch-insert more efficiently), `TABLE` (simulates a sequence with an ordinary
table, rarely used), `AUTO` (let Hibernate pick). `IDENTITY` is simplest and
fine for this project's scale — the batching downside doesn't matter here.

**Important nuance:** it's "next value from an internal counter," not "previous
row's id + 1." Ids are unique and increasing but can have gaps — e.g. a
deleted row or a rolled-back insert permanently skips a number, since the
counter had already advanced. Normal and expected; ids identify rows, they
don't count them.

## `@Column(name = "...")` vs relying on default naming, and `updatable = false`

`@Column(name = "...")` is only needed to override the column name Hibernate
would otherwise pick. Hibernate auto-converts camelCase field names to
snake_case column names by default (`createdAt` → `created_at` automatically),
so writing `name = "created_at"` is technically redundant — it's there purely
as visible documentation of the real column name. Single-word fields like
`name` need no conversion, so no explicit `name = "name"` was written (would
be pure noise). This is a style choice, not a functional requirement.

`updatable = false` is unrelated to naming — it means "never allow an UPDATE
to change this column after INSERT." This is intentional per-field:
`createdAt` has it (a creation timestamp shouldn't change, and indeed
`ReviewTask` has no `setCreatedAt()`), while `name` doesn't (renaming should be
allowed, and `ReviewTask` does have `setName(...)`). Both the DB-level
(`updatable = false`) and Java-level (no setter) protections work together.

## `@ManyToOne(fetch = FetchType.LAZY)` + `@JoinColumn`

`@ManyToOne` describes relationship shape: written on the "many" side (e.g.
`Question` → `Paper`: many questions, one paper). No annotation needed on the
"one" side unless you also want a back-reference collection (skipped for now).

`fetch = FetchType.LAZY` controls *when* the related object loads: LAZY = only
load the related row (e.g. `Paper`) when code actually calls `.getPaper()` and
touches a field on it; EAGER = always load it immediately, even if unused.
LAZY avoids wasted queries. Note: JPA's spec default for `@ManyToOne` is
actually EAGER — we deliberately override it. Gotcha for later: accessing a
lazy field after the DB session already closed throws
`LazyInitializationException`.

`@JoinColumn` (not `@Column`) is used because the field (`Paper paper`) isn't a
plain value — it's a reference to another entity. `@JoinColumn(name =
"paper_id")` tells Hibernate which foreign-key column stores that
relationship. Under the hood it's still just a foreign key number; the
annotation just lets Java code navigate it as `question.getPaper()...` instead
of manually looking up by id.

## `@PrePersist` — why `Paper`/`ReviewTask` have it, `KnowledgePoint` doesn't

`@PrePersist` is a JPA lifecycle callback: marks a method to run automatically
right before Hibernate INSERTs that entity (part of a family — `@PostPersist`,
`@PreUpdate`, `@PreRemove`, `@PostLoad`, etc.).

It's only added where there's an actual field that needs auto-filling at
creation time: `ReviewTask.createdAt` and `Paper.uploadedAt` both have
timestamp columns to set, so both get a `@PrePersist` method. `KnowledgePoint`
(and `Question`) have no timestamp column in their schema at all — nothing to
auto-fill, so no `@PrePersist` method. Not a special exemption, just no need.

## Modeling the `Question` ↔ `KnowledgePoint` many-to-many (`question_tags`)

Decided to keep the many-to-many design from PROJECT.md (a question CAN have
multiple knowledge point tags), using a plain `@ManyToMany` + `@JoinTable`
(no separate `QuestionTag` Java class), unidirectional — only declared on
`Question`, not back-referenced from `KnowledgePoint`, since every query we
need starts from the `Question` side.

```java
@ManyToMany(fetch = FetchType.LAZY)
@JoinTable(name = "question_tags",
        joinColumns = @JoinColumn(name = "question_id"),
        inverseJoinColumns = @JoinColumn(name = "knowledge_point_id"))
private Set<KnowledgePoint> knowledgePoints = new HashSet<>();
```

- `joinColumns` = the FK column pointing back to *this* entity (`Question`);
  `inverseJoinColumns` = the FK column pointing to the *other* entity
  (`KnowledgePoint`). Both must be wrapped in `@JoinColumn(name = "...")` —
  writing a bare column name (no `@JoinColumn(...)`) is a compile error,
  since Java tries to resolve it as a variable, not a string.
- Field type must be a **collection** (`Set`, not a single object reference)
  — a single-object field is what `@ManyToOne` looks like, not `@ManyToMany`.
- `Set` over `List`/`Collection`: prevents duplicate tag pairs at the Java
  level, matching the real-world semantic (tagging the same question with the
  same knowledge point twice is meaningless).
- Always initialize the field (`= new HashSet<>()`), or a new entity's field
  is `null` and calling `.add(...)` on it throws an NPE.

Verified: Hibernate created the exact join table PROJECT.md specifies — just
two FK columns, no separate id, composite primary key:
```sql
create table question_tags (
  question_id bigint not null,
  knowledge_point_id bigint not null,
  primary key (question_id, knowledge_point_id)
)
```
